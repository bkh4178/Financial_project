# AWS를 활용한 AI/ML 협업 개발 환경 구축 가이드

이 문서는 AWS를 처음 사용하는 팀원이 **하나의 AWS 계정 안에서 협업**하며 AI/ML 모델을 개발하는 전체 과정을 안내합니다. 로컬 환경과의 차이점부터 AWS의 핵심 서비스, 실제 계정 설정부터 VS Code 원격 개발 환경 구축까지, 모든 환경 설정을 완료할 수 있도록 구성되었습니다.

---

## 1. AWS란 무엇인가?

AWS(Amazon Web Services)는 아마존닷컴이 제공하는 **클라우드 컴퓨팅 서비스**입니다. 쉽게 말해, 인터넷을 통해 아마존이 소유한 전 세계의 거대한 데이터 센터에 있는 **컴퓨터 자원(서버, 저장 공간 등)을 빌려 쓰는 것**입니다. 우리가 필요할 때마다 고성능 컴퓨터를 즉시 빌려서 사용하고, 사용한 만큼만 비용을 내는 개념입니다.


---

## 2. AWS를 왜 사용해야 하는가? (로컬 개발 vs. AWS)

로컬 환경(개인 노트북, 데스크톱)에서의 개발과 비교했을 때, AWS를 사용하면 AI/ML 개발의 고질적인 문제들을 해결하고 차원이 다른 효율성을 얻을 수 있습니다.

| 항목 | 로컬 환경에서의 개발 | AWS를 이용한 개발 |
| :--- | :--- | :--- |
| **자원 접근성** | 내 컴퓨터의 성능(CPU, RAM, GPU)이 곧 개발의 한계. 수백만 원짜리 고성능 GPU는 매우 비싸고 구하기 어려움. | 클릭 몇 번으로 최신 고성능 GPU 서버(**NVIDIA A10, V100 등**)를 필요한 시간만큼만 빌려 쓸 수 있음. **(자원의 한계 극복)** |
| **환경 통일성** | 팀원마다 OS, 라이브러리 버전(PyTorch, CUDA), 드라이버 등이 달라 "제 컴퓨터에선 되는데..." 문제가 빈번하게 발생. | 모든 팀원이 동일한 클라우드 서버 환경에 접속하여 작업하므로, 버전 충돌이나 환경 차이로 인한 문제가 원천적으로 발생하지 않음. **(완벽한 환경 통일)** |
| **협업 효율성** | 수십~수백 GB의 대용량 데이터셋을 공유하기 위해 외장 하드를 사용하거나, 느린 속도로 파일을 압축/전송해야 함. | 중앙 데이터 저장소(S3)에 데이터를 올려두면, 모든 팀원이 가장 빠른 속도로 데이터에 즉시 접근하여 사용할 수 있음. **(초고속 데이터 공유)** |
| **확장성 및 유연성** | 더 좋은 성능이 필요하면 컴퓨터를 새로 사야 함. 사용하지 않을 때도 컴퓨터는 감가상각되며 전기를 소모. | 간단한 모델 테스트는 저렴한 서버로, 최종 모델 학습은 강력한 서버로 유연하게 변경 가능. 사용하지 않을 때는 **서버를 꺼두어 비용을 완벽하게 절약.** **(유연한 자원 관리)** |
| **재현성 및 관리** | 모델 학습 기록, 실험 결과가 개인 PC에 흩어져 있어 체계적인 관리가 어렵고, 과거의 성공적인 실험을 재현하기 힘듦. | 모든 코드, 데이터, 실험 결과가 클라우드에 중앙 집중식으로 관리됨. **MLOps** 파이프라인을 구축하여 실험을 자동화하고 결과를 추적하기 용이. |

결론적으로 AWS는 **자원의 한계를 없애고, 모든 팀원이 일관된 환경에서 효율적으로 협업하며, 실험 과정을 체계적으로 관리**할 수 있게 해주는 강력한 디지털 작업 공간입니다.

## 3. AWS 주요 서비스 소개

### 직접 다룰 핵심 서비스 4가지

1.  **Amazon S3 (Simple Storage Service)** (알고리즘 개발팀)
    * **설명:** 인터넷 세상의 **'무한한 용량의 외장 하드'**입니다. 파일, 이미지, 동영상, 데이터셋 등 모든 종류의 데이터를 안전하고 저렴하게 저장할 수 있습니다.
    * **사용 목적:** 모든 팀원이 공유할 **대용량 데이터셋**과 학습된 **모델 가중치(weights)**를 저장하는 중앙 데이터 저장소로 사용합니다.

2.  **Amazon EC2 (Elastic Compute Cloud)** (서비스 개발팀)
    * **설명:** 필요에 따라 사양을 자유롭게 조절할 수 있는 **'가상 컴퓨터(서버)'**입니다. Ubuntu, Windows 등 원하는 OS를 설치하고 웹 서버를 구축하는 등 거의 모든 작업을 할 수 있는 가장 기본적인 서비스입니다.
    * **사용 목적:** 최종적으로 완성된 AI 모델을 API 형태로 외부에 제공하거나, 웹 서비스를 배포하는 **메인 서버**로 사용될 수 있습니다.

3.  **Amazon SageMaker** (알고리즘 개발팀)
    * **설명:** AI/ML 모델 개발의 전 과정을 지원하는 **'AI 개발자용 통합 서비스'**입니다. 데이터 전처리, 모델 학습, 배포, 모니터링 기능을 모두 제공하여 개발자가 오직 모델링에만 집중할 수 있도록 돕습니다.
    * **사용 목적:** 이 서비스의 **'노트북 인스턴스'** 기능을 **원격 GPU 개발 서버**로 활용하여, 각자 개인용 개발 환경을 구축하고 모델 학습을 진행합니다.

4.  **Amazon Route 53** (서비스 개발팀)
    * **설명:** `www.google.com`과 같은 사람이 읽기 쉬운 도메인 이름을, 컴퓨터가 이해하는 IP 주소(`172.217.160.142`)로 연결해주는 **'인터넷 세상의 주소록(DNS)'** 서비스입니다.
    * **사용 목적:** EC2 서버에 배포된 우리 서비스를 사용자들이 `www.우리팀프로젝트.com`과 같은 주소로 쉽게 찾아올 수 있도록 연결하는 역할을 합니다.

### 그 외 알아두면 좋은 AI/ML 관련 서비스

* **AWS Glue:** 대규모 데이터셋에 대한 **ETL(추출, 변환, 적재)** 작업을 자동화해주는 서비스입니다. S3에 쌓인 원본 데이터를 분석하기 좋은 형태로 가공할 때 매우 유용합니다.
* **Amazon Athena:** S3에 저장된 데이터 파일을 **표준 SQL 쿼리**를 사용해 직접 분석할 수 있게 해주는 서비스입니다. 데이터를 DB에 옮기지 않고도 빠르게 내용을 확인할 수 있습니다.
* **AWS Lambda:** 서버를 직접 관리할 필요 없이 코드만 업로드하면 알아서 실행해주는 **'서버리스(Serverless)'** 컴퓨팅 서비스입니다. 간단한 모델 추론 API를 만들거나 데이터 처리 파이프라인을 구축할 때 유용합니다.
* **Amazon RDS (Relational Database Service):** MySQL, PostgreSQL 같은 관계형 데이터베이스를 쉽게 설치하고 관리(백업, 업데이트 등)해주는 서비스입니다. 정형 데이터를 다룰 때 S3 대신 사용할 수 있습니다.
* **Amazon ECR (Elastic Container Registry):** 개발 환경을 **'컨테이너(Docker)'** 이미지로 만들어 저장하는 공간입니다. 어떤 컴퓨터에서든 동일한 환경을 100% 재현할 수 있어, 모델 배포 시 환경 불일치 문제를 해결합니다.

## 4. AWS를 통한 환경 구축하기 (A to Z)

### Root 계정과 IAM 계정: 건물주와 세입자

하나의 계정으로 여러 명이 협업하기 때문에, 계정의 역할을 명확히 이해하는 것이 **보안과 비용 관리**의 핵심입니다.

* **Root 계정:**
    * AWS에 처음 가입할 때 사용한 이메일 계정입니다.
    * 계정의 모든 권한을 가진 **절대 관리자**입니다.
    * **역할:** IAM 사용자(세입자) 생성, 결제 관리, 서비스 사용량 제한 설정 등 계정 전체의 관리 작업만 수행합니다. **평소 개발 작업에는 절대 사용하지 않습니다.**
    * **보안:** **반드시 MFA(다단계 인증)를 설정**하여 탈취 위험을 막아야 합니다.

* **IAM 사용자:**
    * Root 계정이 만들어주는 개인별 하위 계정입니다.
    * `S3 사용 가능`, `SageMaker 사용 가능`처럼 **'최소 권한의 원칙'**에 따라 필요한 권한만 부여받습니다.
    * **역할:** 모든 개발자는 자신의 IAM 계정으로만 로그인하여 실제 개발 작업을 수행합니다. 실수로 다른 팀원의 작업을 망치거나, 과도한 비용을 발생시키는 것을 방지합니다.

### [관리자용] 초기 환경 설정

이 과정은 팀의 대표 관리자(Root 계정 소유자) 한 명만 수행하며, 팀원들은 개념만 이해하면 됩니다.

#### **1. 사용자 그룹 및 권한 설정**
1.  Root 계정으로 로그인 후, **IAM** 서비스로 이동합니다.
2.  왼쪽 메뉴에서 **'사용자 그룹'**을 클릭하고 '그룹 생성'을 누릅니다.
3.  그룹 이름(예: `capstone-algorithm`)을 정하고, 아래 **5가지 필수 정책**을 검색하여 체크하고 그룹을 생성합니다.
    * `AmazonS3FullAccess`: S3 버킷 생성 및 파일 업/다운로드 권한
    * `AmazonSageMakerFullAccess`: SageMaker 노트북 인스턴스 생성 및 관리 권한
    * `AmazonEC2FullAccess`: EC2 서버 생성 및 관리 권한
    * `AmazonRoute53FullAccess`: Route53 도메인 연결 권한
    * `AWSSystemsManagerForSSHDocuments`: **VS Code 원격 연결(SSM)**에 반드시 필요한 권한

#### **2. SageMaker 공용 실행 역할 생성**
1.  IAM 서비스 왼쪽 메뉴에서 **'역할'**을 클릭하고 '역할 만들기'를 누릅니다.
2.  신뢰할 수 있는 엔터티 유형에서 **'AWS 서비스'**를 선택합니다.
3.  사용 사례에서 **'SageMaker'**를 선택하고 다음으로 넘어갑니다.
4.  권한 정책에서 `AmazonSageMakerFullAccess`를 검색하여 체크하고 다음으로 넘어갑니다.
5.  역할 이름(예: `SageMaker-Execution-Role-Common`)을 지정하고 역할을 생성합니다. 이 역할은 모든 팀원이 SageMaker 인스턴스를 만들 때 공용으로 사용됩니다.

#### **3. SageMaker 도메인 생성 (최초 1회)**
1.  **SageMaker** 서비스 > **'도메인'** > **'도메인 생성'**을 클릭합니다.
2.  **'빠른 설정'**을 선택하고, 도메인 이름과 기본 사용자 이름을 지정합니다.
3.  네트워크 및 데이터 암호화 등은 기본 설정을 사용하고 도메인을 생성합니다. 이 과정은 10분 정도 소요될 수 있습니다.

#### **4. 팀원용 IAM 계정 생성 및 SageMaker 사용자 프로필 추가**
1.  IAM 서비스 왼쪽 메뉴에서 **'사용자'**를 클릭하고 '사용자 생성'을 누릅니다.
2.  사용자 이름(예: `dev-user1`)을 입력하고 **'AWS Management Console 액세스 제공'**을 체크합니다.
3.  **'IAM 사용자를 생성하고 싶습니다'**를 선택하고, 초기 비밀번호를 설정한 뒤 **'사용자는 다음에 로그인할 때 새 암호를 생성해야 합니다'**를 체크합니다.
4.  권한 설정 단계에서 **'사용자를 그룹에 추가'**를 선택하고, 1단계에서 만든 그룹(`capstone-algorithm`)에 체크합니다.
5.  생성을 완료하고, 생성된 각 팀원에게 아래 **3가지 정보**를 전달합니다.
    * **콘솔 로그인 URL:** `https://<12자리계정ID>.signin.aws.amazon.com/console` 형식의 전용 URL
    * **IAM 사용자 이름** (예: `dev-user1`)
    * **초기 비밀번호**
6. **SageMaker** 서비스 > **'도메인'**에서, 생성된 각 IAM 사용자에 대한 **'사용자 프로필'**을 '사용자 추가' 버튼을 통해 추가합니다.

### [개발자용] 개발 환경 구축 및 사용법

#### **1. S3 버킷 생성 및 폴더 구조화**
1.  전달받은 IAM 계정으로 로그인 후, **S3** 서비스로 이동합니다.
2.  팀의 대표 한 명이 `capstone-algorithm-data`와 같이 팀 전체가 사용할 **공용 버킷**을 하나만 생성합니다. (버킷 이름은 전 세계에서 유일해야 합니다.)
3.  협업을 위해 버킷 안에 다음과 같이 폴더를 만들어 사용합니다.
    * `/shared-data/`: 모든 팀원이 공용으로 사용할 데이터셋을 업로드하는 폴더
    * `/dev-user1/`, `/dev-user2/`: 각자 개인적인 실험 결과나 중간 산출물을 저장하는 개인 폴더
    * `/models/`: 최종 학습이 완료된 모델 파일을 저장하는 폴더

#### **2. SageMaker Studio와 VS Code 원격 연결하기**
1.  **AWS 콘솔**에서 **SageMaker** 서비스 > **'도메인'**으로 이동합니다.
2.  본인의 사용자 프로필 옆에 있는 **'앱 시작(Launch app)' > 'Studio'**를 클릭하여 메인 '로비' 화면을 엽니다.
3.  **(최초 1회)** Studio 로비에서 **'스페이스 생성(Create Space)'**을 클릭하고, 본인만 사용할 **'Private' 스페이스**를 하나 만듭니다. (예: `kijeong-private-space`)
4.  **로컬 VS Code**의 **AWS Toolkit** 패널을 엽니다.
5.  **SAGEMAKER** 섹션에서 방금 만든 본인의 Private 스페이스를 찾아 옆의 **플러그 모양 '연결' 아이콘**을 클릭합니다.
6.  새로운 VS Code 창이 열리며 원격 서버에 연결됩니다. (상태표시줄이 초록색으로 변경)

#### **3. 프로젝트 코드 및 데이터 불러오기**
1.  VS Code 내장 터미널을 엽니다 (`Ctrl` + `~`).
2.  `git clone <GitHub 리포지토리 주소>` 명령어를 직접 실행하여 코드를 가져옵니다.
3.  `aws s3 sync s3://<팀공용버킷이름>/ ./data/` 명령어로 S3에 있는 공용 데이터를 로컬 `data` 폴더로 다운로드합니다.

#### **4. Python 패키지 환경 설정 (Conda 활용)**
팀원 모두가 GPU(CUDA) 버전을 포함하여 100% 동일한 환경을 사용하기 위해 Conda를 활용합니다.

1.  관리자가 미리 프로젝트 폴더에 작성해 둔 **`environment.yml`**과 **`setup_conda_env.sh`** 스크립트를 확인합니다.
2.  터미널에서 아래 명령어를 순서대로 실행하여 **설치 스크립트를 딱 한 번 실행**합니다.
    ```bash
    # 스크립트에 실행 권한 부여
    chmod +x setup_conda_env.sh
    # 설치 스크립트 실행
    ./setup_conda_env.sh
    ```
3.  스크립트 실행이 완료되면, `Ctrl` + `Shift` + `P`를 눌러 **`Python: Select Interpreter`**를 실행하고, 안내된 경로(예: `~/conda/envs/capstone-env/bin/python`)를 지정해줍니다.


#### **5. AI/ML 모델 개발 워크플로우**
1.  **작업 시작:** SageMaker 콘솔에서 **'Studio'를 시작**하고, VS Code로 원격 접속합니다.
2.  **코드 동기화:** VS Code 터미널에서 `git pull`로 최신 코드를 받아옵니다.
3.  **가상 환경 활성화:** 터미널에서 `conda activate capstone-env`를 실행합니다.
4.  **개발 진행:** 브랜치 생성 후 모델 학습, 코드 수정 등의 작업을 진행합니다.
5.  **작업 공유:** 작업이 끝나면 `git add`, `commit`, `push`를 통해 변경된 코드를 GitHub에 공유합니다.
6.  **작업 종료 (매우 중요‼️):** VS Code 또는 Studio 웹 화면에서 실행 중인 **커널(Kernel)과 터미널 세션을 모두 종료(Shutdown)**하여 비용을 절약합니다.

## 5. 비용 관리 및 필수 Best Practice

클라우드는 사용한 만큼 비용을 내는 만큼, 관리하지 않으면 '요금 폭탄'을 맞을 수 있습니다. 아래 수칙을 반드시 지켜주세요.

1.  **쓰지 않는 인스턴스는 즉시 중지(Stop)하기:** SageMaker 인스턴스는 켜져 있는 시간 내내 비용이 발생합니다. 작업이 끝나면 반드시 **'중지'** 버튼을 누르는 것을 습관화해야 합니다.
2.  **AWS Budgets 설정하기:** Root 계정 관리자는 **'결제'** 대시보드 > **'예산(Budgets)'** 메뉴에서 월별 예상 비용 한도를 설정할 수 있습니다. 예를 들어, '월 100달러를 초과하면 이메일 알림 보내기'와 같은 규칙을 만들어 예상치 못한 비용 발생을 막을 수 있습니다.
3.  **CloudWatch 경보 설정:** 특정 인스턴스가 N시간 이상 실행되면 알림을 보내는 **'CloudWatch 경보'**를 설정하여, 실수로 끄지 않은 인스턴스를 찾아낼 수 있습니다.
4.  **올바른 인스턴스 유형 선택:** 처음부터 가장 비싼 GPU 서버를 사용하는 대신, 코드 디버깅이나 간단한 테스트는 저렴한 CPU 인스턴스(`ml.t3.medium`)에서 수행하고, 실제 대규모 학습이 필요할 때만 GPU 인스턴스(`ml.g4dn.xlarge`)를 사용하는 것이 효율적입니다.
