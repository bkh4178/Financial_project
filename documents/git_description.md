# Git Description
깃을 사용하는데 필요한 기초지식을 정리한 문서입니다.

---

### .gitignore

깃를 사용하다보면 종종 필요없는 파일(시스템 파일 등등)이 들어가곤 합니다. 이를 사전에 방지하기 위해서 [gitignore.io](https://www.toptal.com/developers/gitignore)를 활용했습니다. 키워드로는 `windows`,`mac`, `react`, `yarn` 을 입력하여 파일을 생성했습니다. 또한 .env.\*, dist/, .vscode를 추가하였습니다.
> **참고** : 본 프로젝트에서는 github에서 repository를 초기에 생성할 때 옵션으로 제공하는 python용 .gitignore를 사용하였으며, 여기에 추적을 원치 않는 디렉토리(e.g. data/)나 확장자(e.g.*.xlsx, *.csv)를 추가해서 사용합니다.

### git convention

모든 개발자가 커밋 내용을 일관되게 작성할 수 있도록 컨벤션을 정의하였습니다. 이를 통해 작업 내용을 쉽게 파악할 수 있으며, 의도적으로 작업 단위를 분리할 수 있도록 하였습니다. [Conventional Commits](https://www.conventionalcommits.org/)을 기반으로 하였고 세부 내용은 다음과 같습니다.

```text

<type>[optional scope]: <description>

[optional body]

[optional footer(s)]

```

#### type

| 타입       | 의미                                    |
| ---------- | --------------------------------------- |
| `build`    | 빌드 관련 작업 수행 (배포 파일 생성 등) |
| `chore`    | 프로젝트 설정 관련 작업 수행            |
| `ci`       | CI/CD 파이프라인 관련 작업 수행         |
| `docs`     | 문서 작성 또는 수정                     |
| `feat`     | 새로운 기능 추가                        |
| `fix`      | 버그 수정                               |
| `refactor` | 코드 리팩토링                           |
| `style`    | 스타일 변경                             |
| `test`     | 테스트                                  |
| `rename`   | 파일 및 폴더 이름 재설정 및 이동, 삭제  |
| `!HOTFIX`  | 릴리즈 버전 치명적인 버그 수정          |

### git branch 전략

위 작업까지 다 완료했다면 이제 깃을 효율적으로 사용하기 위해서 깃 브랜치 전략에 대해서 정해야합니다. git flow, github flow 등등 여러 방법이 있지만 그 중 git flow를 선택하였습니다. 사실 이것은 여러 인원에서 작업하는 것이 효율적인 방식이기 때문에 맞지 않는 전략이지만 이후 늘어날 것이라고 생각하여 사용하기로 하였습니다.

#### git-flow

해당 프로젝트에서 쓸 브랜치를 정의하겠습니다.

| 부모 브랜치 | 브랜치 명   | 역할                                     | 병합 브랜치  |
| ----------- | ----------- | ---------------------------------------- | ------------ |
|             | `main`      | product 서버에 배포 및 버전 관리         |              |
| `main`      | `dev`       | 각각의 기능 개발이 끝난 후 취합          | `main`       |
| `dev`       | `feature/*` | 기능 개발                                | `dev`        |
| `dev`       | `release`   | 배포 테스트 및 QA 진행                   | `dev`,`main` |
| `main`      | `hotfix`    | product 서버에서 문제가 생겼을 시에 처리 | `dev`,`main` |

### tagging convention

`v<MAJOR>.<MINOR>.<PATCH>` 형식을 준수하고, 프리릴리즈 태그는 `alpha` (내부 테스트), `beta` (공개 테스트), `rc` (릴리즈 전 최종 테스트)로 관리합니다.
태그를 통해 자동 배포를 관리하는 것도 고려하고 있습니다.

## 1. 프로젝트 시작하기 (최초 1회 설정)

가장 먼저 우리 팀의 `algorithm` 프로젝트를 여러분의 로컬 컴퓨터로 가져오는 과정입니다.

### 1-1. 프로젝트 복제 (Clone)

GitHub 레포지토리(저장소)에 있는 모든 코드와 히스토리를 내 컴퓨터로 복제합니다.

```bash
# 1. 터미널(Git Bash, PowerShell 등)을 엽니다.
# 2. 아래 명령어를 입력하여 프로젝트를 복제합니다.
# [레포지토리 주소]는 GitHub에서 제공하는 HTTPS 또는 SSH 주소를 사용합니다.
git clone https://github.com/StatCapstoneDesign/algorithm.git
```

### 1-2. 개인 작업 디렉토리 생성

복제가 완료되면, 프로젝트 폴더로 이동하여 본인의 **영문 이름**으로 된 개인 디렉토리를 생성합니다. 앞으로 여러분의 모든 코드는 이 디렉토리 안에서 작성됩니다.

```bash
# 1. 방금 복제한 프로젝트 폴더로 이동합니다.
cd algorithm

# 2. 본인의 영문 이름으로 폴더를 생성합니다. (예: kijeong)
mkdir kijeong
```

이제 모든 초기 설정이 완료되었습니다!

## 2. 반복적인 개발 워크플로우 (Tuning, Validation, Development)

모델 튜닝, 검증, 신규 모델 개발 등 모든 작업은 아래의 워크플로우를 따릅니다. **절대로 `main` 브랜치에 직접 코드를 추가하거나 수정하지 않습니다.**

### 단계 1: `main` 브랜치 최신 상태로 동기화

새로운 작업을 시작하기 전, 다른 팀원들이 작업해서 `main` 브랜치에 업데이트된 최신 코드를 내 로컬 컴퓨터로 가져옵니다. 이 과정을 생략하면 나중에 코드가 꼬일 수 있으니 **항상 가장 먼저 수행해야 합니다.**

```bash
# 1. main 브랜치로 이동합니다.
git checkout main

# 2. 원격 저장소(GitHub)의 최신 변경사항을 가져와 내 로컬 main 브랜치에 반영합니다.
git pull origin main
```

### 단계 2: 작업 전용 브랜치 생성

하나의 독립된 작업을 수행하기 위한 **개인 작업 공간(브랜치)**을 만듭니다. 브랜치 이름은 어떤 작업인지 명확히 알 수 있도록 아래 규칙에 따라 만들어주세요.

**브랜치 이름 규칙**: `[작업종류]/[작성자]/[상세내용]`

* **작업 종류**: `feature`(기능 개발), `fix`(버그 수정), `docs`(문서 작업) 등
* **작성자**: 본인 영문 이름 (예: `kijeong`)
* **상세 내용**: 작업 내용을 간략하게 설명 (예: `add-tuning-script`)

```bash
# 예시: kijeong이 하이퍼파라미터 튜닝 스크립트를 추가하는 경우
git checkout -b feature/kijeong/add-tuning-script

# 위 명령어는 새로운 브랜치를 생성하고(-b) 그 브랜치로 바로 이동(checkout)하는 명령어입니다.
```

### 단계 3: 코드 작업 및 변경사항 저장 (Add & Commit)

새로 만든 브랜치에서 자유롭게 코드를 작성하고 수정합니다. 작업은 반드시 본인의 **개인 디렉토리**(`kijeong/`) 안에서 진행해야 합니다.

작업이 어느 정도 일단락되면, 변경된 내용을 **의미 있는 단위**로 묶어 저장(`commit`)합니다.

```bash
# 1. 내가 변경한 모든 파일들을 스테이징(Staging) 영역으로 이동시킵니다.
# 스테이징: "이 파일들을 곧 저장할 겁니다"라고 Git에게 알려주는 단계
git add .

# 2. 스테이징된 파일들을 메시지와 함께 저장(commit)합니다.
# 커밋 메시지는 어떤 작업을 했는지 명확하게 작성해야 합니다.
git commit -m "feat(tuning): kijeong 하이퍼파라미터 튜닝 스크립트 추가"
```

> **💡 좋은 커밋 메시지란?**
>
> 다른 사람이 메시지만 보고도 어떤 변경사항이 있었는지 짐작할 수 있어야 합니다.
> * **Good**: `feat(tft_pess): Add learning rate scheduler`
> * **Bad**: `코드 수정`, `버그 해결`

### 단계 4: 원격 저장소에 업로드 (Push)

로컬 컴퓨터에 저장한 커밋들을 이제 GitHub 원격 저장소로 업로드할 차례입니다. 이 작업을 해야 다른 팀원들이 여러분의 코드를 보고 리뷰할 수 있습니다.

```bash
# 현재 브랜치의 변경사항을 원격 저장소(origin)에 동일한 이름의 브랜치로 업로드합니다.
git push origin feature/kijeong/add-tuning-script
```

## 3. Pull Request (PR) 작성 및 코드 리뷰

`Push`를 완료하면, 여러분이 작업한 브랜치의 내용을 `main` 브랜치에 합쳐달라고 공식적으로 요청하는 **Pull Request(PR)**를 생성할 수 있습니다.

우리 팀은 개인별 디렉토리에서 작업하므로 코드 충돌 가능성은 거의 없습니다. 하지만 **PR은 다음과 같은 중요한 목적을 가집니다.**

* **코드 리뷰**: 다른 팀원들이 코드를 보며 더 좋은 방향을 제시하거나 실수를 바로잡아 줄 수 있습니다.
* **지식 공유**: "아, 이 사람은 이런 방식으로 문제를 해결했구나!" 하고 서로 배우는 기회가 됩니다.
* **작업 기록 관리**: 어떤 기능이 왜, 어떻게 추가되었는지 기록으로 남길 수 있습니다.

### PR 생성 및 작성 가이드

1.  `Push`를 완료하고 GitHub 레포지토리에 접속하면, 노란색 배경의 **"Compare & pull request"** 버튼이 나타납니다. 이 버튼을 클릭하여 PR 생성을 시작합니다.
2.  PR 제목과 내용을 아래 템플릿에 맞춰 상세하게 작성합니다.

```markdown
### PR 제목 (Title)

[작업종류] [작성자] 작업 내용 요약
예: `feat(tuning): kijeong 하이퍼파라미터 튜닝 스크립트 및 결과 추가`

---

### 📝 작업 요약 (Summary)

이번 PR에서 어떤 작업을 왜 수행했는지 간략하게 설명해주세요.

> 예: TFT 모델의 최적 하이퍼파라미터를 찾기 위해 `src` 모듈을 활용한 튜닝 스크립트를 작성했습니다.

###  perubahan 변경 내용 (Changes)

구체적으로 어떤 파일이 추가/변경되었는지 목록을 작성합니다.

-   `kijeong/tuning/run_tuning.py`: 하이퍼파라미터 튜닝을 실행하는 메인 스크립트
-   `kijeong/results/tuning_summary.csv`: Grid Search 튜닝 결과 요약 파일

### 📊 검증 결과 (Validation)

작업 결과를 확인할 수 있는 수치나 스크린샷, `wandb` 링크 등을 첨부해주세요.

-   기존 모델 Test IC: 0.03
-   튜닝 후 모델 Test IC: 0.035
-   WandB 대시보드 링크: [링크 바로가기](...)

### ✅ 체크리스트 (Checklist)

PR을 올리기 전 스스로 점검하는 항목입니다.

-   [x] `main` 브랜치의 최신 코드를 반영했습니다. (`git pull origin main`)
-   [x] 작업 브랜치에 불필요한 파일이 포함되지 않았습니다.
-   [x] 내가 작업한 코드가 오류 없이 실행되는 것을 확인했습니다.
```

PR이 생성되면 다른 팀원들이 코드를 확인하고 댓글(코멘트)을 남길 수 있습니다. 리뷰가 완료되고 승인(Approve)되면, `Merge pull request` 버튼을 눌러 `main` 브랜치에 최종적으로 코드를 반영합니다.

## 4. 실습 1단계: 로컬에서 코드 만들고 Push 하기

이제 이론을 배웠으니, 직접 따라하며 워크플로우를 익혀봅시다!

**목표**: 본인 개인 디렉토리 안에 `practice` 폴더를 만들고, `hello.txt` 파일을 추가하여 GitHub에 Push하기

```bash
# 1. main 브랜치로 이동 및 최신화
git checkout main
git pull origin main

# 2. 실습용 브랜치 생성 (본인 이름으로 변경!)
git checkout -b feature/kijeong/practice-file

# 3. 내 개인 디렉토리로 이동 (본인 이름으로 변경!)
cd kijeong

# 4. 실습용 폴더 및 파일 생성
mkdir practice
echo "Hello GitHub! This is my first contribution." > practice/hello.txt

# 5. 프로젝트 최상위 디렉토리로 다시 이동
cd ..

# 6. 변경사항 확인 (practice/hello.txt가 추가된 것을 볼 수 있음)
git status

# 7. 변경사항 저장
git add .
git commit -m "feat(practice): kijeong 실습 파일 추가"

# 8. 원격 저장소에 업로드 (본인 이름으로 변경!)
git push origin feature/kijeong/practice-file
```

## 5. 실습 2단계: GitHub에서 Pull Request 생성하기

이제 터미널에서의 작업은 끝났습니다. GitHub 웹사이트로 이동하여 방금 올린 코드를 `main` 브랜치에 합쳐달라고 요청해봅시다.

1.  **GitHub 레포지토리 방문**: `push`를 완료하고 GitHub의 `algorithm` 레포지토리 페이지로 접속하면, 노란색 배경의 안내창이 보입니다. **[Compare & pull request]** 버튼을 클릭하세요.
    

2.  **PR 내용 작성**: PR 생성 화면으로 이동됩니다. `base: main` ← `compare: feature/kijeong/practice-file` 로 설정되었는지 확인하세요. 이는 "내가 작업한 브랜치의 내용을 main 브랜치에 합치겠다"는 의미입니다. 이제 위에서 배운 **PR 템플릿**에 맞춰 제목과 내용을 정성껏 작성합니다.

    **✍️ 실습용 PR 작성 예시**
    ````markdown
    ### PR 제목 (Title)
    feat(practice): kijeong 실습 파일 추가

    ---
    ### 📝 작업 요약 (Summary)
    GitHub 협업 가이드 실습을 위해 `hello.txt` 파일을 추가했습니다.

    ### 🔁 변경 내용 (Changes)
    - `kijeong/practice/hello.txt`: 실습용 텍스트 파일 생성

    ### 📊 검증 결과 (Validation)
    - 해당 없음

    ### ✅ 체크리스트 (Checklist)
    - [x] `main` 브랜치의 최신 코드를 반영했습니다. (`git pull origin main`)
    - [x] 작업 브랜치에 불필요한 파일이 포함되지 않았습니다.
    - [x] 내가 작업한 코드가 오류 없이 실행되는 것을 확인했습니다.
    ````

3.  **PR 생성 완료**: 내용 작성이 끝났다면, 오른쪽 하단의 초록색 **[Create pull request]** 버튼을 클릭하여 PR 생성을 완료합니다. 이제 팀원들에게 리뷰를 요청할 준비가 되었습니다!

## 6. PR 리뷰, 병합(Merge) 및 브랜치 정리

PR이 생성되면 팀원들이 코드를 검토하고 피드백을 줍니다. 모든 논의가 끝나고 코드를 `main`에 반영해도 좋다는 합의가 이루어지면, PR을 병합하고 사용했던 브랜치를 정리합니다.

### 6-1. 코드 리뷰 (Code Review)

* 팀원들은 PR의 `Files changed` 탭에서 변경된 코드를 확인하고 의견이 있을 경우 댓글을 남깁니다.
* 리뷰 요청을 받은 사람은 피드백을 반영하여 코드를 수정하고, 다시 `add` -> `commit` -> `push` 과정을 거쳐 PR에 변경사항을 업데이트합니다.

### 6-2. PR 병합 (Merge)

* 리뷰가 끝나고 모두가 동의하면, 관리자 또는 PR을 생성한 사람이 **[Merge pull request]** 버튼을 클릭하여 `main` 브랜치에 코드를 최종 반영합니다.
    

### 6-3. 브랜치 정리 (Branch Cleanup)

성공적으로 병합된 브랜치는 더 이상 필요 없으므로, 원격 저장소와 로컬 저장소에서 모두 삭제하여 프로젝트를 깔끔하게 유지합니다.

1.  **원격(GitHub) 브랜치 삭제**: Merge 후 나타나는 **[Delete branch]** 버튼을 클릭하여 GitHub에 있는 브랜치를 삭제합니다.

2.  **로컬 브랜치 삭제**: 이제 내 컴퓨터에 남아있는 브랜치를 정리할 차례입니다.

    ```bash
    # 1. 다시 main 브랜치로 돌아옵니다.
    git checkout main

    # 2. main 브랜치에 방금 병합된 최신 내용을 반영합니다. (매우 중요!)
    git pull origin main

    # 3. 사용이 끝난 로컬 브랜치를 삭제합니다. (본인 브랜치 이름으로 변경!)
    git branch -d feature/kijeong/practice-file
    ```

이것으로 하나의 작업 사이클이 모두 끝났습니다! 새로운 작업을 시작하려면 다시 [2단계: 반복적인 개발 워크플로우](#2-반복적인-개발-워크플로우)로 돌아가 동일한 과정을 반복하면 됩니다.

## 7. 유용한 Git Branch 명령어 모음

알아두면 협업이 훨씬 편해지는 Branch 관련 명령어들입니다.

**브랜치 목록 보기**

```bash
# 내 컴퓨터(로컬)의 브랜치 목록을 봅니다. (*가 현재 브랜치)
git branch

# 원격 저장소(GitHub)의 브랜치까지 모두 봅니다.
git branch -a
```

**브랜치 생성 및 이동**
```bash
# 새로운 브랜치를 만들고 바로 그 브랜치로 이동합니다.
git checkout -b [새 브랜치 이름]

# 이미 있는 다른 브랜치로 이동합니다.
git checkout [이동할 브랜치 이름]
```

**브랜치 삭제**
```bash
# 병합이 완료된 로컬 브랜치를 삭제합니다. (안전)
git branch -d [삭제할 로컬 브랜치 이름]

# 병합되지 않은 브랜치도 강제로 삭제합니다. (주의! 작업 내용이 사라질 수 있음)
git branch -D [삭제할 로컬 브랜치 이름]

# 원격 저장소(GitHub)의 브랜치를 삭제합니다.
git push origin --delete [삭제할 원격 브랜치 이름]
```
